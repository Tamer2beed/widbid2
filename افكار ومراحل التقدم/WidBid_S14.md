## خارطة الطريق — المرحلة التالية

### 🔴 أولوية عالية: نظام الصلاحيات (S15)

**المشكلة الحالية:** لا يوجد فصل حقيقي بين الصلاحيات — أي مستخدم يعرف البنية قد يصل لميزات أعلى من رتبته.

**المطلوب تنفيذه:**

#### 1. Middleware الخادم
```
server/middleware/rankGuard.js
- يتحقق من JWT وrank قبل كل socket event
- canActOn(actorRank, targetRank, minRequired)
- يرفض الطلبات التي تتجاوز الصلاحية
```

#### 2. Socket Event Validation
```
كل event يتحقق من:
- هل المستخدم مسجّل دخول؟ (token صالح)
- هل rank المستخدم يكفي للعملية؟
- هل الهدف (target) أقل رتبة من الفاعل؟
```

#### 3. Client-side Guard
```
كل زر/ميزة في الواجهة:
- تُخفى إذا rank المستخدم أقل من المطلوب
- حتى لو أرسل request يدوياً → الخادم يرفضه
```

#### 4. جدول الصلاحيات الكامل
```
رتبة 100 (زائر):    كتابة رسائل، طلب مايك
رتبة 200 (عضو):     + رفع يد، إرسال صور
رتبة 300 (محمي):    + اسم محمي (لا أحد يغيره)
رتبة 400 (ملكي):    + أفضلية في طابور المايك
رتبة 500 (مشرف):    + كتم/طرد rank 100-400
رتبة 600 (سوبر):    + حظر + تعديل ترحيب
رتبة 700 (ماستر):   + ترقية/تخفيض 200-500
رتبة 800 (سوبر م):  + ترقية/تخفيض حتى 600
رتبة 900 (روت):     + تعديل إعدادات الغرفة
رتبة 1000 (س.روت):  + ترقية/تخفيض حتى 800
رتبة 1100 (أونر):   + إدارة كاملة للغرفة
رتبة 1200 (س.أونر): كل الصلاحيات
```

#### 5. التحقق المزدوج (Double Verification)
```
Client → يتحقق قبل إرسال الطلب
Server → يتحقق عند استقبال الطلب
الاثنان يجب أن يتطابقا
```
